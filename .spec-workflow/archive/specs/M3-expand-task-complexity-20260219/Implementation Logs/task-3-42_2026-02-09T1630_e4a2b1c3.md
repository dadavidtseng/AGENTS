### Key Design Decisions

1. **Debouncing by questId:fileType** — Coalesces rapid successive writes (e.g., quest creation writes metadata + tasks + requirements) into single events per file type per quest.

2. **awaitWriteFinish** — Chokidar's built-in write stabilization (200ms threshold, 50ms poll) prevents reading partially-written files.

3. **File type mapping** — Maps recognized filenames (metadata.json, tasks.json, etc.) to WebSocket event types (quest.updated, task.completed, approval.requested).

4. **Graceful degradation** — If QUEST_DATA_PATH is not set, file watcher is disabled with a warning. Server still functions normally via API polling.

### Environment Variables

#### QUEST_DATA_PATH
- **Purpose:** Absolute path to mcp-server-quest .quest-data directory
- **Required:** No (file watcher disabled if not set)
- **Example:** C:/GitHub/mcp-server-quest/.quest-data

### Integrations

#### Integration
- **Description:** File watcher monitors .quest-data/quests/{questId}/ for file changes and broadcasts WebSocket events to dashboard clients
- **Data Flow:** File change → chokidar → debounce → readFile → parse → broadcastEvent(wsEventName, payload) → WebSocket clients
- **Event Mapping:** metadata.json → quest.updated, tasks.json → task.completed, approval-history.json → approval.requested
