## Artifacts

### Modules

#### events/broadcast.ts (new)
- **Location:** src/events/broadcast.ts
- **Purpose:** Standalone event broadcasting module, decoupled from any HTTP/WS server
- **Key changes from old dashboard/events.ts:**
  1. **Synchronous** — old functions were `async` due to dynamic `import()` of dashboard server; new functions are sync
  2. **Pluggable handler** — `setBroadcastHandler(fn)` allows any consumer to register a broadcast sink
  3. **No-op by default** — when no handler is registered, broadcasts are silent (correct for MCP-only path)
  4. **Typed event payloads** — exported interfaces for all 8 event types
- **Exports:** `setBroadcastHandler`, `broadcastQuestCreated`, `broadcastQuestUpdated`, `broadcastApprovalRequested`, `broadcastApprovalDecision`, `broadcastTaskAssigned`, `broadcastTaskStatusChanged`, `broadcastAgentRegistered`, `broadcastAgentStatusChanged`

### Design Decisions

1. **Pluggable handler vs hard removal** — Rather than removing broadcast calls entirely, we kept them as no-ops with a pluggable handler. This preserves the event architecture for future consumers (e.g. mcp-client-quest's file watcher could register a handler that writes event files).
2. **Await compatibility** — Callers still use `await broadcastXxx()`. Since the new functions are synchronous, `await` on a non-Promise is a no-op in JS. This avoids unnecessary churn across 11 files.
3. **Config cleanup** — Removed `dashboardPort` and `dashboardHost` from Config interface since they're no longer used by any code path.
